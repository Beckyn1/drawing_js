Sprint 8

For sprint 8, my main goal was to make a brush and along the way, I decided to redo the star shape.

I had no idea where to start off with when making a brush function, so I had to do some research. I found this person on YouTube called "Franks Laboratory", where he was showing how the javascript stars worked mathematically as well as how to make a star brush. From this video, I decided to redo my star shape as I liked his version of doing it better than the one I currently had, and I also derived my brush from his star brush.

I made a test file to try out his technique (as can be seen in the video). Though his code was a bit different to the code I used in my drawing program, I figured that it was essentially just a star function being called under the mMove event. So, I did a lot of trial and error and found out that I could just insert a circle function into mMove and a brush would form. However, at first because I used a function not an object, the brush would only go on the canvas and not the drawing space. I then had to make it an object but also then had to add a ctx.clip to the drawing space to cut out the brush strokes that went over the boundaries. Once I had figured it out, however, I was able to experiment with more brushes. I found some code for how to make a spray brush, and I adapted that to my program. I also experimented with different shape types and made some quite unique brushes.

Stars

My first design for the star was using line to and move to repetitively to go to certain points from the mouse. This, however, was not working, as can be seen in the photo. The code was also very repetitive and inflexible.

The second design was by using trig functions and loops to draw and connect lines. This did work and worked a lot better than the first attempt, but, however was not very flexible in terms of if I wanted to add more star points later as there was a lot of fixed values (the design is essentially a pentagon with spikes, so you have to change values for both attributes to add more points(increase number of loops doesn't do anything). The star also didn't look like how I wanted it to as it was quite thing and on a weird angle, and I also wanted the star to not change width as it changes size (so it always stays a perfectly shaped star).

The third design was done by having a concept of two circles, so there is an inner and outer radius. The idea is that there is a line that draws from the centre to the outer radius and then to the inner radius, and then the shape rotates to draw the next point depending on the number of points used (rotates it based on pi/number of points as it splits up a circle). I like this design better as it makes it very easy to change the number of points a star has, as I can just change the n value when calling the function. It also increases the size of the star as a whole instead of changing width and height individually.

Relevant Implications Summary

Future proofing

Future proofing is where you design your program to be easily updatable in the future. This means that the program is easy to add new features to and changes to different parts of the program are easy to change. In my program I have addressed future proofing by making different objects/classes/functions and then calling them through a main object. This way, I can easily individually work on each component and then put it all together in one place. Being able to work on each component individually is good for future proofing because it means I can easily add, delete or change things without disrupting the entire program in future sprints. For example, when I wanted to add a new shape, I just made a new shape function and then called it via the main controlobject object and then for every shape after that, I just repeat the same process and the main overall program will be unaffected if something goes wrong with an individual shape. My program is also easy for another developer to work on and improve. My code is well commented and has docstrings at the top of every function. The docstrings have a description of what the function does and they also say what the parameters refer to in the program. This makes it easier for the developer to see what each function does. I have also made it easier for another developer to work on and improve by separating parts of the program into separate files and connecting them with the index file. Separating files into their main functions makes it easier for another developer to see which part controls what, so that the main file doesn't become too congested and hard to read/work with. By using a sprint based design and pushing to github after each sprint, I have also addressed future proofing. This method ensures that if anything goes wrong in a sprint, I still have a backup from the previous sprint, so I don't have to start from nothing. As I make a new sprint everytime I make a big change, this helps with future proofing as I can make lots of changes without potentially ruining my entire program (as opposed to one sprint).

Usability

Usability is where the program is easy to use for the user. This means that the program gives informative feedback, and minimises errors the user can make. In my program, I have designed the buttons so that they change colour when selected to show that the button that is clicked is selected. The button also changes colour when the mouse hovers over it. As well as this, for the options where 2 actions are required (eg choosing a colour and a shape), 2 buttons are shown to be selected at once. The layout of my drawing program is also easy for the user to use. This is because I have separated all the functions into blocks that are grouped by function (eg shapes are in one section, lines and line width in another section). As similar actions are next to each other, it makes it easier for the user as they can easily find and associate similar actions. I have also made the layout easy to use by using blocks and big text boxes. Each shape function has the name of the shape on it etc, which makes it easy for the user to associate the action with the word. The drawing area is also large and clearly separated from the buttons, which makes it easy for the user to distinguish.

Functionality

Functionality is making sure the program works well and runs smoothly. This means that the program does not crash, totals get added properly and pizzas are added properly. In my program, I have addressed functionality by systematically testing my program after each sprint, as well as regularly testing my program throughout the addition of a new feature. As I have thoroughly tested my program, I have been able to spot most (if not all) errors and fixed them so that the program runs smoothly. As far as I am concerned, my program does not crash, so the user should not have any serious issues whilst using the program. I have also been checking for errors via the console log, as this shows an error message when something doesn't work. For my final sprint, my console log was blank, so no errors were detected. When I was figuring out how to make a brush I also made a separate test file, just for testing brush codes. I had to do research on how to make a brush, and so I followed a video which used slightly different methods than the rest of my drawing program. To figure out how to alter the code to fit my program and also so I could thoroughly test the brush code and make sure it worked how I wanted (as well as experiment with different brush types), I made a separate testing file. I also tested in both safari and chrome, as users may have different browser preferences and I needed to make sure the program worked well in both conditions. As well as my personal testing, I have also had multiple other people test my program, some with digital technologies experience and some with none. By testing my program with people who had digital technologies knowledge, it helped me to spot and fix technical errors with my program that I may have not noticed. Testing my program with people that were not knowledgeable with digital technologies was also helpful as they told me what parts of the program were easy to understand and what parts were slightly confusing. An example is the font/font size which I changed to make it easier to read.
